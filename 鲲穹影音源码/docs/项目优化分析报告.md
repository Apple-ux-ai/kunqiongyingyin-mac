# 鲲穹影音 - 项目优化分析报告

基于对 `鲲穹影音源码` 的阅读与检索，从**安全、性能、可维护性、体验、架构与运行时**五方面整理可优化点与建议。  
（第二次分析补充了「渲染进程与 Electron 环境判断」等运行时问题。）

---

## 〇、项目结构概览

| 类型 | 路径/说明 |
|------|-----------|
| 主进程入口 | `main.js`（窗口、IPC、解码器、关灯、更新、登录 IPC） |
| 渲染进程入口 | `index.html` → `js/app.js`（ES Module）→ `script.js`（BaofengPlayer） |
| 预加载 | `preload.js`（contextBridge 暴露 `electronAPI`、有限 `ipcRenderer`） |
| 主逻辑 | `script.js`（约 7800+ 行，播放/列表/字幕/设置/截图等） |
| 模块 | `js/modules/`：config、logger、i18n、player-engine、ui-manager、auth-manager（渲染端）、resume/playlist/dlna/hotkeys mixin |
| 主进程模块 | `decoder.js`、`logger.js`、`auth-manager.js`（主进程）、`update-manager.js`、`ad-manager.js`（渲染层）、`login-manager.js`（渲染层） |
| 多语言 | `js/locales/*.js` + `i18n.js`，支持 20+ 语言 |
| 构建 | Electron + electron-builder，Win x64，NSIS/portable，extraResources 含 ffmpeg/ffprobe、updater.exe |

## 一、安全相关

### 1. 主进程密钥硬编码（高优先级）

- **位置**: `main.js` 第 21 行  
- **问题**: `SECRET_KEY` 直接写在源码中，一旦仓库或构建产物泄露，登录签名可被伪造。  
- **建议**:  
  - 使用环境变量（如 `process.env.KQ_SECRET_KEY`）或构建时注入。  
  - 若必须放仓库，至少不要提交到公开仓库，并配合密钥轮换策略。

### 2. 打开外部链接未校验（高优先级）

- **位置**: `main.js` 中 `ipcMain.on('open-external', ...)`，直接 `shell.openExternal(url)`。  
- **问题**: 渲染进程（含广告 API 返回的 `target_url`）可传入任意协议，如 `javascript:`、`file:`，存在钓鱼或本地文件泄露风险。  
- **建议**: 在主进程中对 `url` 做白名单校验，仅允许 `http:` / `https:`，否则不调用 `shell.openExternal`。

### 3. 使用 eval 解析帧率（中优先级）

- **位置**: `decoder.js` 第 138 行，`fps: eval(videoStream.r_frame_rate)`。  
- **问题**: `r_frame_rate` 来自 ffprobe（如 `"30/1"`、`"24000/1001"`），虽多为数字运算，但用 `eval` 有潜在风险且不利于维护。  
- **建议**: 改为安全解析，例如将 `"30/1"` 按 `/` 分割后 `num/den` 计算 fps。

### 4. 广告/API 数据直接插入 DOM（中优先级）

- **位置**: `ad-manager.js` 中 `renderTitleAd` / `renderPlaylistAd` 使用 `innerHTML` 和 `ad.adv_url`、`ad.target_url`。  
- **问题**: 若接口被篡改或返回恶意内容，可能造成 XSS。  
- **建议**:  
  - 对 `adv_url`、`target_url` 做格式校验（URL 白名单、协议限制）。  
  - 图片用 `setAttribute('src', ...)` 并校验 URL；跳转链接仅用于 `openExternal` 且在主进程校验（见上一条）。

## 二、性能与资源

### 5. 单一大文件 script.js（中优先级）

- **现状**: `script.js` 体量很大（约 7800+ 行），包含播放器主逻辑、UI、列表、字幕、设置等。  
- **问题**: 首屏解析/编译时间长，协作与维护成本高，按需加载困难。  
- **建议**:  
  - 已存在 `js/modules/`（如 `playlist-mixin.js`、`hotkeys-mixin.js` 等），可继续把「字幕」「设置」「画面/音频调节」等拆成独立模块，主文件只做组合与初始化。  
  - 对非首屏必需的面板（如设置、DLNA）可考虑动态 import，减少首包体积。

### 6. 第三方资源外链（中优先级）

- **位置**: `index.html` 中 Font Awesome：`https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css`。  
- **问题**: 依赖外网，离线或 CDN 故障时图标缺失；无 SRI，存在被篡改风险。  
- **建议**:  
  - 将字体/图标资源打包到本地（如 `assets/fonts/`），或使用 npm 包本地引用。  
  - 若继续用 CDN，建议加上 `integrity` 和 `crossorigin`。

### 7. 日志与 console（低优先级）

- **现状**: 多处使用 `console.log/warn/error`，且部分与 `logger` 并存。  
- **建议**:  
  - 主进程统一走 `logger`，避免直接 `console`，便于生产环境关闭或分级。  
  - 渲染进程可封装一层“前端 logger”，在开发环境才输出到 console，生产可关闭或上报。

### 8. 更新接口使用 HTTP（低优先级）

- **位置**: `update-manager.js` 中 `updateUrl = 'http://software.kunqiongai.com:8000/...'`。  
- **问题**: 未加密，在不可信网络下可能被劫持或篡改更新包。  
- **建议**: 若服务端支持，改为 HTTPS。

## 三、可维护性与一致性

### 9. 默认播放器检测的 appId 不一致（高优先级）

- **位置**: `main.js` 第 218 行，检查注册表时使用 `com.kunqiong.aiplayer` 和 `kunqiong-ai-player`。  
- **问题**: `package.json` 中 `appId` 为 `com.kq.player.v113.ultimate`，与检测逻辑不一致，会导致“设为默认”后仍显示“非默认”。  
- **建议**: 将检测用的字符串改为与 `package.json` 的 `build.appId` 一致（如 `com.kq.player.v113.ultimate`），或从配置/package 读取同一 appId。

### 10. preload 中 ipcRenderer 暴露范围（中优先级）

- **位置**: `preload.js` 中 `exposeInMainWorld('ipcRenderer', { invoke: ... })`，仅允许少量 channel。  
- **问题**: 若渲染进程需要调用其它合法 channel，容易漏加，且与主进程实际提供的 channel 可能不同步。  
- **建议**:  
  - 主进程维护一份“允许渲染进程调用的 channel 列表”，preload 只允许该列表；或  
  - 不再暴露通用 `invoke`，改为为每个功能暴露具名 API（如 `electronAPI.selectDirectory`），与当前 `electronAPI` 风格统一。

### 11. 软件编号/ID 分散（低优先级）

- **现状**: `SOFT_NUMBER` / `softwareId` 在 `auth-manager.js`（10019）、`ad-manager.js`（10019）、`update-manager.js`（10007）等多处硬编码。  
- **建议**: 在 `config.js` 或单例配置中统一维护（如 `SOFT_NUMBER`、`UPDATE_SOFTWARE_ID`），避免改版本/产品时漏改。

## 四、体验与健壮性

### 12. 错误与加载状态（低优先级）

- **建议**:  
  - 转码/加载失败时，除 console/logger 外，在 UI 上给出明确提示（如“该格式需要转码，转码失败：原因”）。  
  - 网络请求（广告、更新、登录）失败时，可考虑重试或“稍后重试”的入口。

### 13. 关灯模式与多显示器（低优先级）

- **位置**: `main.js` 中 `createLightsOffWindow` 使用 `primaryDisplay`。  
- **问题**: 多显示器时仅主屏变黑，副屏可能仍亮。  
- **建议**: 若产品需要，可改为遍历 `screen.getAllDisplays()` 为每个显示器创建一个遮罩窗口。

### 14. 打包与构建（低优先级）

- **现状**: `package.json` 的 `files` 已排除 `logs`、`*.log`、`dist*/**` 等，避免把无关文件打进包。  
- **建议**:  
  - 构建前自动执行一次 `decoder.cleanupTempFiles()` 或清理临时目录，避免误打包临时文件。  
  - 若存在敏感配置或密钥，确保未通过 `files` 或 `extraResources` 打进 asar。

## 六、架构与运行时（第二次分析补充）

### 15. 渲染进程中使用 require('electron') / @electron/remote（高优先级）

- **位置**: `script.js` 多处：
  - 约 2598–2601 行：`toggleCompactMode()` 内 `require('electron')`、`require('electron').remote`、`require('@electron/remote').getCurrentWindow()`；
  - 约 2408–2411 行：截图保存备用分支里 `require('electron').remote?.dialog`、`require('@electron/remote')`；
  - 约 5646–5647 行：截图路径初始化里 `require('electron').remote`、`require('@electron/remote')`（已包在 `if (typeof require !== 'undefined')` 内）。
- **问题**: 主进程已设置 `nodeIntegration: false`、`contextIsolation: true`，渲染进程（包括通过 `js/app.js` 加载的 `script.js`）中 **`require` 不可用**。上述代码一旦执行会抛出 `ReferenceError: require is not defined`。且项目未依赖 `@electron/remote`，即便在旧版 Electron 中也会报错。
- **影响**:
  - **toggleCompactMode()**：当前未见 UI 调用，属死代码；若将来被菜单/按钮触发，将直接崩溃。
  - **截图保存**：主路径已使用 `window.electronAPI.showSaveDialog` 并 `return`，备用分支仅在无 electronAPI 时执行；在现有 preload 下不会走到 require，但逻辑仍依赖“有 require = Electron”，见下条。
  - **截图路径初始化**：仅在 `typeof require !== 'undefined'` 时用 `app.getPath('pictures')`；在真实 Electron 渲染进程中 require 为 undefined，导致从不走该分支，默认“图片”路径可能依赖 index 里写死的 `C:\Users\win10\Pictures` 或 localStorage。

**建议**:
- 不再用 `typeof require !== 'undefined'` 判断“是否 Electron”，改为 **`window.electronAPI` 或 `window.electronAPI?.selectDirectory` 等是否存在**。
- 需要“当前窗口”“系统对话框”“app.getPath”时，一律通过 **preload 暴露的 IPC** 由主进程完成（例如已有 `selectDirectory`、`showSaveDialog`、可新增 `getPath('pictures')`、`getWindowBounds`/`setWindowBounds` 等），渲染进程只调 `window.electronAPI.xxx`。
- 删除或重写 `toggleCompactMode()`，改用 IPC 通知主进程改变窗口尺寸；删除所有渲染进程内对 `require('electron')`、`require('@electron/remote')` 的引用。

### 16. 截图保存逻辑中“Electron 环境”判断错误（高优先级）

- **位置**: `script.js` 中 `saveScreenshot()`：整个“Electron 保存/选择路径”逻辑包在 `if (typeof require !== 'undefined') { ... } else { 浏览器下载 }`。
- **问题**: 在标准 Electron 渲染进程（无 require）下，**永远进入 else**，被当成“浏览器环境”，用 `<a download>` 下载，而不会走 `window.electronAPI.showSaveDialog`。结果是：在 Electron 里首次截图或未设路径时，无法弹出“另存为”对话框，只能触发浏览器式下载（行为可能不符合预期）。
- **建议**: 用 `window.electronAPI` 判断是否在 Electron 中，例如：
  - `if (window.electronAPI?.showSaveDialog) { /* 使用 showSaveDialog + IPC 写文件 */ } else { /* 浏览器 <a download> */ }`  
  并将“选择路径 / 写文件”全部放在该分支内，不再依赖 `require`。

### 17. UIManager 与 script.js 中选择器不一致（低优先级）

- **位置**: `js/modules/ui-manager.js` 的 `cacheElements()` 使用如 `querySelector('.play-pause-btn')`、`'.stop-btn'`、`'.progress-slider'` 等；`index.html` 中实际 id 为 `playBtn`、`stopBtn`、`progressBarContainer` 等，类名也不完全一致。
- **问题**: 若播放器主界面仍由 `script.js` 直接通过 getElementById 等绑定，UIManager 的 elements 可能多为 null，其 `updatePlayPauseBtn`、`updateProgress` 等可能不生效或报错。
- **建议**: 确认当前播放器是否以 script.js 的 DOM 绑定为准；若是，则让 UIManager 的选择器与 index.html 实际 id/class 对齐，或改为仅负责部分子模块，避免与 script.js 重复绑定同一控件。

### 18. 配置与版本分散（低优先级）

- **位置**: `js/modules/config.js` 中为 `VERSION: 'v5.76.0209.1111'`，`package.json` 为 `"version": "1.1.1"`；软件编号等仍分散在 auth-manager、ad-manager、update-manager。
- **建议**: 版本号尽量单一来源（如从 package.json 或构建时注入）；软件编号、更新地址等可收拢到 config.js 或环境配置，便于发布与多产品复用。

## 七、小结与修复顺序

当前项目结构清晰，Electron 与 preload 使用规范（如 contextIsolation、禁用 nodeIntegration）。  
**已在前次修改中完成**：默认播放器 appId 与 package 一致、open-external 仅允许 http(s)、decoder 帧率安全解析。

**建议优先处理**：  
1）**渲染进程去掉所有 require('electron') / @electron/remote**，改为通过 preload 暴露的 IPC/API（含窗口尺寸、getPath、对话框等）；  
2）**截图保存与“Electron 环境”判断**改为基于 `window.electronAPI`，保证在 Electron 下能正常弹出“另存为”并保存；  
3）**SECRET_KEY** 改为环境变量或构建注入；  
4）**广告/API 的 URL 校验**与 innerHTML 安全（见上文）。

其余项（script.js 拆模块、Font Awesome 本地化、更新 HTTPS、日志统一、UIManager 选择器、配置集中）可按排期逐步落地。
